## 1. 현황 요약

FitFlow PoC를 인수하여 **5개 보고된 결함을 모두 진단·수정**하고, **재발 방지를 위한 회귀 테스트 19건**을 추가했습니다. 전체 테스트(기존 8 + 신규 15) 모두 통과 상태이며, Docker 없이 SQLite + pytest로 수십 초 내 회귀 점검이 가능합니다.

수정한 결함 외에도 인수 과정에서 **데이터 정합성 오염**(시드의 의도적 케이스), **timezone-aware 컬럼 부재**, **PT 잔여 계산 로직의 3중 중복**, **PRD vs 비즈니스 규칙 간 환불 수식 충돌** 등 4건의 추가 이슈를 식별했고, 각각 보고서 §4·§5에 분류해두었습니다.

PoC 단계 시스템 치고는 비즈니스 규칙 문서(`docs/business_rules.md`)와 시드 데이터에 의도적 엣지 케이스가 잘 심어져 있어 *진단 가능성*이 높았습니다. 다만 비즈니스 로직이 라우터·서비스·챗봇 3계층에 흩어져 있어 **같은 결함이 여러 위치에 동시에 존재**하는 패턴을 반복적으로 마주쳤습니다(B3가 대표적).

## 2. 결함 분석

전체 진단 로그는 `diagnosis_log.md`에 상세히 기록했고, 본 섹션은 CTO 의사결정에 필요한 요약입니다.

| ID | 위치 | Root cause (한 문장) | 영향 |
|----|------|---------------------|------|
| **B1** | `services/membership_calculator.py` | `avg_visits_per_month=0` 시 `float('inf')` 폴백 + 매직 넘버 `+10`으로 PT/회원권 도메인을 한 함수에 혼재 | 회원 상세 응답에 `"Infinity"` 문자열 노출 → 프론트 표기 깨짐 |
| **B2** | 동상 | `calculate_expiry_date()`가 `FreezePeriod`를 조회하지 않음 (BR-2.2 위반) | 동결 회원 만료일이 실제보다 앞당겨 표시 → 조기 만료 처리 위험 |
| **B3** | `routers/pt_sessions.py`, `routers/members.py`, `services/chatbot.py` | 사용량 집계 시 `package_id` 필터·`is_trial` 제외 누락 (BR-3.2/BR-5.3 위반) | 다중 패키지·체험 보유 회원의 잔여 횟수 부정확 → 환불·정산 분쟁 위험 |
| **B4** | `services/stats_aggregator.py` | naive UTC datetime으로 월 경계 비교 (BR-5.1 KST 위반) | 월말/월초 가입자 집계 누락·중복 → 대시보드 신뢰도 저하 |
| **B5** | `services/pt_matching.py` | `matched[0]` 반환 + 가용량 미체크 + 부분 매칭 (BR-4.1/BR-4.2 위반) | 인기 트레이너 과부하, 가득 찬 트레이너 추천 → 서비스 품질 저하 |

수정 방식의 공통 원칙:
1. **증상이 아닌 root cause 수정**. B1의 `"Infinity"` 캐스팅 우회는 제거하고 함수 책임을 재정의.
2. **BR과 시스템 컨벤션이 충돌하면 시스템 컨벤션 일관성을 우선**. B2의 동결 일수 계산이 대표 사례 (BR 본문 "6일" vs 결과값 "2/5" → exclusive 5일 채택).
3. **같은 결함이 여러 위치에 있으면 모두 수정**. B3는 3개 파일을 동시 수정해야 결함이 진짜 사라집니다.

bdd4c78 fix(B4): 월별 통계 집계 시 KST 경계를 UTC로 변환

## 3. 검증 전략

상세 설계는 `verification_design.md` 참고. 핵심 정책:

- **재현 → 수정 → 회귀 사이클**: 각 결함마다 (1) 실패하는 테스트를 먼저 쓰고, (2) 수정으로 통과시키고, (3) 인접 경계값 테스트를 추가. 코드 표면이 아니라 *결함 자체*를 잠그는 것이 목적.
- **API 레벨 단언**: 함수 내부 구현이 바뀌어도 외부 응답 계약이 유지되는지를 검증. 리팩터링 안전성 확보.
- **결정적·재현 가능**: random 셔플 회피, 시간 의존은 상대 일수로 표현, 격리된 SQLite로 매 테스트 초기화.
- **인프라 가벼움**: Docker/Postgres 부팅 비용 없이 CI에서 수십 초 회귀.

| 항목 | 수치 |
|------|------|
| 전체 테스트 | 23 (기존 8 + 결함 15) |
| 결함당 테스트 평균 | 3 (정상/비정상/경계) |
| 평균 실행 시간 | 약 5초 |
| 외부 의존성 | 없음 (SQLite in-process) |

**한계**: B3 정합성 오염 케이스(`_seed_data_inconsistencies`), 환불 위약금 충돌, 동시성, Postgres 고유 동작은 본 스위트 범위 외. §5 로드맵에 분리.

## 4. 기술 부채

진단 과정에서 발견한 추가 이슈를 우선순위·근거와 함께 정리합니다.

### 4.1. 데이터 정합성 오염 (Critical)
- **현상**: `_seed_data_inconsistencies()`가 의도적으로 심어둔 "유령 활성 회원"(member.status=active이지만 회원권 만료), "팬텀 비활성"(반대), "PT 사용량 > 패키지 총 횟수" 케이스가 운영 환경에서도 동일 패턴으로 발생할 가능성 높음.
- **근거**: handoff 노트의 "회원 상태 필드가 실제 회원권 상태와 다른 경우가 있다"는 인계, 시드 코드 line 372~440이 이를 명시적으로 재현.
- **위험**: 회원 상태 기반 통계·자동화·환불이 잘못된 데이터로 동작.

### 4.2. timezone-aware 컬럼 부재 (High)
- **현상**: 모든 `DateTime` 컬럼이 naive. KST 변환을 애플리케이션 레이어에서 *매 쿼리마다* 수동으로 수행해야 함.
- **근거**: B4 수정 시 month-range 헬퍼를 추가했으나, 이는 본질적으로 임시방편. Postgres `timestamptz` + Python `aware datetime`이 정도(正道).
- **위험**: 새 통계 쿼리가 추가될 때마다 동일 결함이 재생산될 수 있음. DST가 없는 KST는 완화 요인이지만 일반화는 불가.

### 4.3. 비즈니스 로직 다중 위치 중복 (Medium)
- **현상**: PT 잔여 계산 로직이 `routers/pt_sessions.py`, `routers/members.py`, `services/chatbot.py` 3곳에 거의 동일한 형태로 존재. B3 수정은 3곳을 모두 손대야 했음.
- **근거**: 결함 재발의 가장 큰 원인은 "한 곳을 고치고 다른 곳을 잊는 것".
- **권고**: `services/pt_calculator.py`로 일원화, 라우터·챗봇은 호출만.

### 4.4. PRD vs 비즈니스 규칙 충돌 (Medium)
- **현상**: 환불 위약금 수식이 PRD 4.1 ("위약금 10%")과 BR-2.4 ("0.9 곱셈" = 위약금 10%)는 동일하나, PRD §3.2 "환불 금액 = 잔여 횟수 x 단가"(PT)와 BR-2.4 "잔여 횟수 x 단가 x 0.9"는 다름. handoff에서도 "기억 가물가물"로 미해결.
- **현재 코드**: `services/refund.py`는 회원권 환불만 구현, PT 환불 미구현.
- **권고**: PM 확정 후 BR 기준으로 통일. 본 과제 범위 외라 미수정.

### 4.5. 라우터 레벨 증상 우회 패턴 (Low)
- **현상**: B1 수정 전 `members.py:128-129`처럼 라우터가 서비스 결과의 `inf`/`nan`을 캐스팅해서 우회하는 패턴 발견.
- **위험**: root cause를 가려서 원인 파악을 지연시킴. 코드 리뷰 가이드에 "라우터에서 직렬화/방어 처리만, 비즈니스 로직 우회 금지" 항목 추가 권고.

## 5. 개선 제안 (단기·중기 로드맵)

### 단기 (1~2 sprint)
1. **데이터 정합성 점검 잡 도입** — 4.1 해결.
   - `members.status`와 활성 회원권 보유 여부 일치 검증 배치 (cron 일 1회).
   - PT 패키지별 사용량 ≤ 총 횟수 검증, 위반 시 알림.
2. **PT 잔여 계산 로직 일원화** — 4.3 해결. `services/pt_calculator.py` 추가, 3곳 호출자 단순화.
3. **환불 정책 PM 확정** — 4.4 해결. PT 환불 함수 구현 + 테스트.
4. **PR 체크리스트에 "BR 인용 + 회귀 테스트" 의무화** — 결함 재발 방지의 프로세스 측면.

### 중기 (1~2 분기)
5. **timezone-aware 마이그레이션** — 4.2 해결.
   - `DateTime` → `DateTime(timezone=True)` (Postgres `timestamptz`).
   - 기존 데이터를 UTC 가정으로 캐스팅.
   - 애플리케이션 코드를 aware datetime으로 정리, B4식 수동 변환 제거.
6. **챗봇 의미 매칭 개선** — Phase 3 선택 항목과 연결. BR-6.2 유의어 처리. LLM 기반으로 가도 좋음 (Anthropic API 등).
7. **트레이너 부하 분산 전략 검토** — B5 수정은 결정적 정렬을 채택했으나 "공정한 분배"가 진짜 목표라면 라운드로빈/할당 큐 기반 분배를 별도 설계 필요.
8. **테스트 커버리지 측정 도입** — 현재는 결함 중심 회귀 위주. 전반적 라인/브랜치 커버리지 가시화.

## 6. AI 활용 소감

본 과제는 **Claude Code (Opus 4.7)** 와의 페어 작업으로 진행했습니다. 전체 대화·세션 로그는 `ai_logs/`에 보존되어 있습니다.

### 활용 방식
- 코드베이스 탐색·요약은 AI에 위임하여 인수 시간을 단축.
- 각 결함마다 (1) AI가 root cause 후보와 수정안 2~3개를 제시 → (2) 본인이 BR·시스템 컨벤션과 대조하여 채택안을 결정 → (3) AI가 코드·테스트 작성 → (4) 본인이 결과 검증 및 커밋 메시지 검토. 즉 *판단은 본인, 실행은 AI*가 기본 분담.
- B2 수정안 비교에서 "함수 시그니처에 `freeze_periods` 주입 vs `db` 주입" 같은 트레이드오프는 본인이 의식적으로 결정 — DI 스타일이 단위 테스트·재사용성에 유리하다는 판단.

### AI를 *그대로 따르지 않은* 사례
- B2 동결 일수 컨벤션에서 BR 본문 ("6일")과 계산 결과 ("2/5") 충돌이 있었을 때, AI 초안은 BR 본문을 무비판적으로 따라 6일 더하는 안을 제시. 본인이 시스템 다른 곳(`memberships.py`의 freeze 검증)이 exclusive 5일을 사용함을 확인하고 "시스템 일관성 우선" 결정으로 전환.
- B5 정렬 정책에서 AI는 random 셔플도 옵션으로 제시. 본인이 *테스트 재현성*과 *부하 분산은 random이 아닌 `current_clients` 갱신이 본질*이라는 근거로 결정적 정렬 채택.

### 한계 인식
- AI 출력은 *그럴듯한 코드*를 빠르게 생성하지만, BR·기존 코드와의 *일관성* 판단은 사람이 직접 해야 함을 반복 확인.
- 결정적이지 않은 조언(예: "이 부분은 기술 부채로 분류 가능")은 본인의 우선순위 판단이 필요. 본 보고서 §4·§5의 우선순위 표는 AI 의견을 참고하되 본인 기준으로 정렬한 결과.
