# FitFlow 백엔드 엔지니어 실무 과제

안녕하세요, 지원자님.

이 과제는 시험이 아닙니다. **함께 일할 수 있는지 서로 확인하는 과정**입니다.
정답을 맞히는 것보다, 문제를 어떻게 파악하고 어떤 판단을 내리는지가 더 중요합니다.
편하게 시작해 주세요.

---

## 과제 배경

**FitFlow**는 수도권 5개 센터를 운영하는 피트니스 센터 체인입니다.
회원 관리, 회원권, PT 세션, 장비 관리를 통합하는 백오피스 시스템의 PoC(Proof of Concept)를 개발 중이었으나, 담당 개발자(Claude Code를 활용하여 개발)가 퇴사하면서 인수인계가 필요한 상황입니다.

지원자님이 이 프로젝트를 인수받아 **결함을 진단·수정**하고, **검증 체계를 설계**하며, **CTO에게 보고할 종합 보고서를 작성**하는 것이 과제의 핵심입니다.

---

## 시간 안내

| 항목 | 내용 |
|------|------|
| **권장 소요 시간** | 6~10시간 |
| **상한** | **10시간 초과 금지** |
| **제출 기한** | 과제 발송일로부터 **7일** |

> **다 끝내는 것이 목표가 아닙니다.**
> 제한된 시간 안에서 우선순위를 정하고, 합리적으로 판단하는 과정을 보여 주세요.
> 미완성이더라도 **과정 기록이 있으면 부분 점수를 부여**합니다.

### 시간 관리 팁

- **Phase 1 + Phase 2에 전체 시간의 약 70%를 배분**하는 것을 권장합니다.
- 한 결함에 **40분 이상 소요**될 경우, 현재까지의 분석 내용을 기록하고 다음 결함으로 넘어가세요.
- 남은 시간에 다시 돌아오는 전략이 효과적입니다.

---

## AI 도구 사용 안내

본 과제는 **실무와 동일한 조건**에서 수행합니다.
**AI 도구(Claude Code, GitHub Copilot 등) 사용을 권장합니다.**

- AI 사용 이력(프롬프트, 대화 로그 등)을 제출하시면 **긍정적으로만 반영**합니다.
- AI 사용 이력 제출로 인한 **감점은 절대 없습니다.**
- AI를 어떻게 활용했는지 자체가 실무 역량의 일부로 평가됩니다.

> **중요:** AI 도구의 출력을 그대로 제출하는 것이 아니라, AI의 결과를 **검증하고 판단한 과정**이 평가의 핵심입니다. 기술면접에서 진단 과정과 의사결정 근거를 직접 설명해 주셔야 합니다.

## 환경 설정 (Phase 0)

> 예상 소요: **30~60분**

프로젝트를 로컬에 구동하고 전체 구조를 파악하는 단계입니다.

### Docker 환경 실행

```bash
docker compose up -d
```

### 트러블슈팅

| 증상 | 확인 방법 |
|------|-----------|
| 컨테이너가 올라오지 않음 | `docker compose logs` 로 에러 메시지 확인 |
| 포트 충돌 | `docker compose ps` 로 포트 매핑 확인 후 `.env` 에서 포트 변경 |
| DB 연결 실패 | `docker compose exec db psql -U fitflow -d fitflow` 로 직접 접속 테스트 |
| 의존성 설치 오류 | 컨테이너 내부에서 `pip install -r requirements.txt` 재실행 |

### 환경 확인 방법

모든 컨테이너가 정상 동작하면 아래 API가 응답해야 합니다.

```bash
# 헬스체크
curl http://localhost:8000/api/health

# 회원 목록 조회 (데이터가 반환되면 정상)
curl http://localhost:8000/api/members
```

## Phase 1: 결함 진단 + 수정 [필수]

아래 5개 증상이 보고되었습니다. 각 결함을 진단하고 수정해 주세요.

### 버그 목록

| ID | 증상 | 확인 방법 | 기대 결과 |
|----|------|-----------|-----------|
| **B1** | 회원권 예상 소진일이 `Infinity`로 표시됨 | `GET /api/members/{id}/detail` (장기 미방문 회원 조회) | `estimated_exhaustion_days`가 유한한 숫자로 반환 |
| **B2** | 동결 회원의 만료일이 올바르게 계산되지 않음 | `GET /api/members/{id}/detail` (동결 이력이 있는 회원 조회) | `expiry_date`가 동결 기간만큼 연장되어 반환 |
| **B3** | PT 잔여 횟수가 정확하지 않음 | `GET /api/pt/remaining/{member_id}` | 무료 체험과 유료 세션이 정확히 구분되어 계산 |
| **B4** | 대시보드 신규 가입 수치가 실제 회원 목록과 불일치 | `GET /api/analytics/new-members?year=2026&month=1` | 해당 월 회원 목록의 실제 수와 일치 |
| **B5** | PT 매칭 추천 시 특정 트레이너만 반복 편중 | `GET /api/matching/recommend/{member_id}` | 조건에 맞는 트레이너가 다양하게 추천 |

### 진행 방법

1. 위 API를 **직접 호출**하여 증상을 확인합니다.
2. 코드를 추적하여 근본 원인(root cause)을 파악합니다.
3. 수정 후 동일 API로 결과를 검증합니다.
4. 각 결함의 진단 과정을 `diagnosis_log.md`에 아래 형식으로 기록합니다.

### diagnosis_log.md 작성 형식

각 결함(B1~B5)에 대해 아래 항목을 기록해 주세요:

```markdown
## B1: [버그 제목]

### 1. 증상 확인
- 호출한 API: (실제 curl 명령어 또는 요청)
- 응답 결과: (실제 응답 데이터 붙여넣기)
- 이 값이 왜 문제인가: (비즈니스 관점에서 설명)

### 2. 원인 추적 과정
- 가설 1: (무엇을 의심했는가) → 확인 결과: (맞았는가/틀렸는가) → 근거
- 가설 2: (다른 가능성) → 확인 결과 → 근거
- (최소 2개 이상의 가설을 기록해 주세요)

### 3. 근본 원인
- 파일: (경로)
- 위치: (함수명 또는 라인)
- 원인: (왜 이 코드가 문제인지 설명)

### 4. 수정 내용
- 변경 전: (해당 코드)
- 변경 후: (수정한 코드)
- 이렇게 수정한 이유:

### 5. 수정 검증
- 수정 후 동일 API 호출 결과: (응답 데이터 붙여넣기)
- 기대 결과와 일치 여부:
```

> **참고:** 이 형식은 가이드라인입니다. 더 나은 구조가 있다면 자유롭게 변경해도 됩니다. 핵심은 **"과정"이 기록**되어 있는 것입니다.

## Phase 2: 검증 체계 설계 [필수]

수정한 결함이 재발하지 않도록 검증 체계를 설계하고 구현합니다.

### 최소 요건

- **각 결함(B1~B5)마다 최소 1개의 pytest 테스트**를 작성합니다.
- 테스트는 `tests/` 디렉토리에 위치시켜 주세요.
- `pytest`로 실행 시 전체 통과(green)되어야 합니다.

### 권장 설계 (가산점)

더 견고한 검증 체계를 설계하면 높이 평가합니다.

- **분류 테스트**: 정상 케이스 / 비정상 케이스 구분
- **경계값 테스트**: 0, 1, 최댓값 등 경계 조건 검증
- **회귀 방지 테스트**: 동일 결함이 다른 조건에서 재발하지 않는지 확인

검증 설계의 의도와 구조를 `verification_design.md`에 정리해 주세요.

## Phase 4: 종합 보고서 [필수]

CTO에게 보고하는 형식으로 `report.md`를 작성합니다.
아래 6개 항목을 포함해 주세요.

1. **현황 요약** — 인수받은 시스템의 전반적 상태
2. **결함 분석** — 각 버그의 근본 원인과 수정 내용
3. **검증 전략** — 설계한 테스트의 구조와 커버리지
4. **기술 부채** — 발견한 추가 이슈나 개선 포인트
5. **개선 제안** — 단기/중기 개선 로드맵
6. **AI 활용 소감** (해당 시) — AI 도구를 어떻게 활용했는지
